## 17. Remaining risks

1. **Ordering coupling between `assemble_considered` and
   `_select_recommended_experiments`.** The PUBLISH branch now
   filters `considered` AFTER experiment selection. If a future
   refactor reorders the calls (selector before
   `assemble_considered`), the filter will be a no-op and the
   invariant could re-fire. Mitigation: the filter is bracketed by
   a 12-line comment explaining the constraint, and
   `_assert_role_uniqueness(out)` runs at the end of the branch.
   Any ordering regression is caught by the assertion.

2. **`recommended_experiments` list iteration order.** The
   experiment selector returns a deterministic list (sorted by
   `(-audience_size, play_id)`). The B6 filter builds a set of
   `play_id`s from that list, so list ordering does not affect the
   filter outcome.

3. **AOV definition drift.** The fixture's `Recommended Experiment`
   and `Recommended Now` opportunity-context blocks both use
   `aligned[L28]['aov']` (per B1.5). A future change to
   `_resolve_aov_for_context` (Phase 5.1 contract risk noted in
   `memory.md`) will shift the addressable-value strings and
   force a fixture refresh. The byte-stable snapshot test surfaces
   this drift loudly.

4. **Suite-order env contamination.** `WINDOW_POLICY=auto` is now
   pinned in the B6 env_overrides, but other env keys in `.env`
   could similarly leak into harness runs if the harness is
   invoked from a polluted `os.environ`. Mitigation: the harness
   already pops `VERTICAL_MODE` / `VERTICAL`; if a future B6 sibling
   regression targets a different env-driven dimension, that key
   must be pinned in the same way.

5. **Fixture refresh discipline.** The `byte-stable snapshot` test
   makes any silent regression on the rendered briefing.html fail
   loudly. Conversely, an intentional renderer change (e.g.
   restructuring CSS, adding new copy, renaming a section) MUST
   be accompanied by an explicit fixture refresh in the same
   commit. This is documented in the failure message
   (`If the drift is intentional, refresh the fixture at ...`).

6. **Pinned `decision_state == publish` reflects current Phase 5
   gating.** A future tightening of the directional pathway gate
   (e.g. a higher consistency threshold across L28 / L56 / L90)
   could push `first_to_second_purchase` below the publish bar on
   the same fixture, flipping the run to ABSTAIN_SOFT. That is a
   real regression to surface, but it is also a possible
   intentional change. The test will fail loudly with the
   receipts-side state assertion; refresh requires both the
   gating change and a deliberate fixture re-pin.

7. **The `decide.py` filter is PUBLISH-only.** ABSTAIN_SOFT B3
   routing and ABSTAIN_HARD do not exhibit the same overlap
   because both set `recommended_experiments=[]`, so the same
   filter would also be a no-op there. If a future change ever
   populates `recommended_experiments` under abstain (which would
   itself violate the contract), the missing filter on those
   branches would trip `_assert_role_uniqueness`. The defensive
   contract holds.
